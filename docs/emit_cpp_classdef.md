# Plan: Translate MATLAB `classdef` to Real C++ Classes

## Status

Open. Picks up from commit `76dfa03` (pure-read inlining for EmitC).

## Goal

Make `-emit-cpp` produce idiomatic C++ for MATLAB `classdef` programs.
Specifically, replace the current "wrapper-over-runtime-hash" output with
real C++ classes that have named member fields and translated method
bodies — eliminating the `matlab_obj_get_f64` / `matlab_obj_set_f64`
runtime calls for scalar properties.

## Current state (after `76dfa03`)

`examples/bank_account.cpp` emits:

```cpp
// User-defined classes (wrappers over the flat ClassName__method functions).
class BankAccount {
public:
  void *_impl;
  BankAccount(double a0, double a1) : _impl(BankAccount__BankAccount(a0, a1)) {}
  void deposit(double a1) { BankAccount__deposit(_impl, a1); }
  ...
};

int main() {
  void* acc = BankAccount__BankAccount(1001.0, 500.0);
  std::cout << matlab_obj_get_f64(acc, (void*)__matlab_str_f0, 2) << '\n';  // acc.Id
  BankAccount__deposit(acc, 250.0);
  ...
}

static void* BankAccount__BankAccount(double id, double bal) {
  void* v0 = matlab_obj_new(1);
  matlab_obj_set_f64(v0, (void*)__matlab_str_f0, 2, id);
  matlab_obj_set_f64(v0, (void*)__matlab_str_f1, 7, bal);
  return v0;
}

static double BankAccount__eq(void* a, void* b) {
  return (matlab_obj_get_f64(a, (void*)__matlab_str_f0, 2) ==
          matlab_obj_get_f64(b, (void*)__matlab_str_f0, 2)) ? 1.0 : 0.0;
}
```

Two problems user flagged:

1. The wrapper class is dead code. `main` calls flat functions
   (`BankAccount__BankAccount(...)`) and runtime helpers
   (`matlab_obj_get_f64(...)`) directly. Variables in `main` are
   `void*`, not `BankAccount`.
2. Even inside the flat functions, every scalar property access goes
   through a string-keyed runtime hash (`matlab_obj_set_f64(obj, "Id",
   2, id)`). For a known scalar `double Id;` field this is purely
   wasted indirection.

## Target output

```cpp
// Generated by matlabc -emit-cpp. Do not edit.
#include <stdint.h>
#include <iostream>

// A small classdef example: ...

class BankAccount {
public:
  double Id;
  double Balance;

  BankAccount() = default;
  BankAccount(double id, double bal) : Id(id), Balance(bal) {}

  void deposit(double amt) { Balance = Balance + amt; }
  void withdraw(double amt) { Balance = Balance - amt; }
  double get_Overdrawn() const { return (Balance < 0.0) ? 1.0 : 0.0; }
  double operator==(const BankAccount &other) const {
    return (Id == other.Id) ? 1.0 : 0.0;
  }
};

class Savings : public BankAccount {
public:
  double Rate;

  Savings() = default;
  Savings(double id, double bal, double rate) {
    Id = id;
    Balance = bal;
    Rate = rate;
  }
  double interest() const { return Balance * Rate; }
};

int main() {
  BankAccount acc(1001.0, 500.0);
  std::cout << acc.Id << '\n';
  std::cout << acc.Balance << '\n';
  std::cout << acc.get_Overdrawn() << '\n';

  acc.deposit(250.0);
  std::cout << acc.Balance << '\n';

  acc.withdraw(1000.0);
  std::cout << acc.Balance << '\n';
  std::cout << acc.get_Overdrawn() << '\n';

  std::cout << (acc == BankAccount(1001.0, 0.0)) << '\n';
  std::cout << Savings(2000.0, 1000.0, 0.05).interest() << '\n';
  return 0;
}
```

No `_impl`, no runtime hash, no flat `BankAccount__BankAccount` /
`BankAccount__eq` definitions. The C++ classes are first-class and `main`
uses them directly. This mirrors what `-emit-python` already does
(commit `5adf4e4` — classdef → Python class blocks).

## Implementation plan

All work is in `lib/MLIR/Passes/EmitC.cpp` plus a small amount of test
golden regeneration. Designed in 7 phases for incremental commits, but
phases 1–5 must land together because they're tightly coupled.

### Phase 1 — Property discovery (~50 lines)

New struct + member fields:

```cpp
struct CppClassDef {
  std::string Super;
  std::vector<std::string> Properties;        // own + inherited, encounter order
  std::vector<mlir::func::FuncOp> Ctors;
  std::vector<mlir::func::FuncOp> Methods;    // non-ctor
};

llvm::StringMap<CppClassDef> Classes;
llvm::StringSet<> ClassMethodFuncs;           // skip flat emission for these
```

New method `Emitter::collectClassdefs(ModuleOp M)` walks every
`func.func` with `matlab.class_name`, groups by class, records the
super class. For each method body, walks every
`matlab_obj_set_f64(_, "Field", _, _)` and
`matlab_obj_get_f64(_, "Field", _)` LLVM call op, extracts the literal
field name from operand 1 (a string-global addressof), validates it as
a C identifier (and not a C++ keyword), dedupes into
`Classes[X].Properties`. Inherited properties are computed at emission
time as `parent.Properties ⊆ child.Properties`.

### Phase 2 — Class block emission (~100 lines)

Replace the existing wrapper-class generation block (currently at
EmitC.cpp ~L1828, the `// User-defined classes (wrappers over the flat ClassName__method functions).` section) with:

```cpp
void Emitter::emitCppClass(StringRef Name, const CppClassDef &CD) {
  // Determine "own" properties (this class but not in parent chain).
  StringSet<> InheritedProps = collectInheritedProperties(CD.Super);
  vector<StringRef> Own;
  for (auto &P : CD.Properties) if (!InheritedProps.count(P)) Own.push_back(P);

  OS << "class " << Name;
  if (!CD.Super.empty()) OS << " : public " << CD.Super;
  OS << " {\npublic:\n";
  for (auto &P : Own) OS << "  double " << P << ";\n";
  OS << "  " << Name << "() = default;\n";
  for (auto F : CD.Ctors)  emitCppMethod(F, Name);
  for (auto F : CD.Methods) emitCppMethod(F, Name);
  OS << "};\n\n";
}
```

### Phase 3 — Method body translation (~100 lines)

`Emitter::emitCppMethod(func::FuncOp F, StringRef Name)` reuses the
existing per-function setup (NextId, scans, computeInlines), then sets
two new state flags:

```cpp
InClassMethodBody = true;
ClassMethodSelf = <first param for non-ctor; obj_new result for ctor>;
ClassMethodClassName = Name;
```

For ctors: walk the body once, find the `matlab_obj_new` call, bind its
result to "this" in `Names`, add it to `SuppressedOps`, find the
trailing `func.return %obj_new_result` and suppress that too. Then
emit the signature `Name(args)` and the body via `emitRegion`.

For non-ctor methods: bind `arg0` to "this" (or skip entirely — leave
it to the rewrite to consume), emit signature
`<RetTy> <method_name>(<params from arg1 onward>)`, then body. For
binary operators (`eq`, `plus`, etc.) the second arg is `other`.
Get-properties (`matlab.method_name = "get.X"`) emit the body for
property `X` (no params after `self`). Static methods skip the `this`
binding.

### Phase 4 — `obj_get/set_f64` rewrite at the call site (~30 lines)

In `Emitter::emitOp(LLVM::CallOp)` (statement form) and
`buildInlineExpr(LLVM::CallOp)` (inline-expression form), check before
the existing call-emission code:

```cpp
if (InClassMethodBody && Call.getCallee() == "matlab_obj_get_f64" &&
    Call.getOperand(0) == ClassMethodSelf)
  -> emit just the field name (`Balance`)

if (InClassMethodBody && Call.getCallee() == "matlab_obj_set_f64" &&
    Call.getOperand(0) == ClassMethodSelf)
  -> emit `Field = <expr>;` (statement only)
```

Outside a class method body, the rewrite still fires when the receiver
has a tracked `ClassValueType` (Phase 6). Field name comes from the
string-global addressof at operand 1.

### Phase 5 — Skip flat-function emission for class methods (~5 lines)

In the run() loop that emits free `func::FuncOp` bodies:

```cpp
if (ClassMethodFuncs.count(F.getSymName())) continue;
```

The wrapper class block (Phase 2) emits each method's body inside the
class instead.

### Phase 6 — Track class types through the IR (~50 lines)

Two new maps:

```cpp
DenseMap<Value, std::string> ClassValueType;       // SSA value -> class name
DenseMap<Operation *, std::string> ClassAllocaType; // alloca -> class name
```

Populated:

- When a `LLVM::CallOp` with a class-ctor callee emits a result, mark
  `ClassValueType[result] = ClassName`.
- When a `LLVM::StoreOp` stores a class-typed value into an alloca,
  mark `ClassAllocaType[alloca] = ClassName`.
- When a `LLVM::LoadOp` loads from a class-tracked alloca, mark
  `ClassValueType[load.result] = the alloca's class`.

### Phase 7 — Variable-declaration types + main rewriting (~80 lines)

When emitting an alloca in `ClassAllocaType`, declare it as
`<ClassName> name;` instead of `void* name;`.

When emitting a `LLVM::CallOp` whose result is in `ClassValueType`,
declare the local with the class type:
`<ClassName> <name> = <ctor-call-rewritten>;`.

Add `Emitter::tryRewriteAsClassCall` that resolves the callee to its
classdef metadata and emits one of:

- ctor: `ClassName(args)`
- regular method: `args[0].method_name(args[1:])`
- get-property: `args[0].PropName` (no parens)
- operator: `args[0] OP args[1]`
- static: `ClassName::method_name(args)`

Wired into both `buildInlineExpr` and the statement-form `emitOp`. The
existing `tryEmitIOSubstitution` and other rewrites stay ahead of it
in the dispatch order so we don't lose `disp` substitutions.

## Edge cases

- **Inherited property handling.** `Savings` collects properties
  `{Id, Balance, Rate}` but only `Rate` is its own — `Id` / `Balance`
  come from `BankAccount`. The Phase 2 emission filters via
  `InheritedProps`, otherwise the C++ class would re-declare members
  and shadow the base class's storage.
- **Ctor super-call detection.** `Savings::Savings(id, bal, rate)`
  could in principle emit `: BankAccount(id, bal), Rate(rate) {}` if we
  detect the leading "Id = id; Balance = bal" pattern. For the first
  cut, body assignments (`{ Id = id; Balance = bal; Rate = rate; }`)
  are simpler and correct — `BankAccount() = default;` initializes the
  parent, then we overwrite. Improve this in a follow-up.
- **`nargin == N` static dispatch.** The frontend wraps ctor / method
  bodies in `if (nargin == N)` checks. With `2.0 == 2.0` already folded
  to live-branch-only (commit `e01be51` and the static-cond fold in
  EmitPython — already mirrored in EmitC's existing logic), this stays
  invisible at the source layer.
- **`Dependent` properties (`get.Overdrawn`).** The frontend names them
  `matlab.method_name = "get.X"`. Phase 3 emits them as plain getter
  methods (`double get_Overdrawn() const { ... }`). Phase 7 rewrites
  call-site reads `obj.Overdrawn` to method calls `obj.get_Overdrawn()`
  in `main` — or, with effort, to `@property`-style C++ via `__attribute__((const))` — but the explicit method form is clearer.
- **Static methods.** `matlab.method_kind = "static"`. Emit
  `static <RetTy> method_name(args)` inside the class; rewrite call
  sites to `ClassName::method_name(args)`. No `this` binding.
- **Operator return types.** `operator==` etc. return `double` in our
  current ABI (1.0 / 0.0). C++ canonical signature returns `bool`; we
  match the runtime's ABI to keep numeric chaining safe with MATLAB
  source like `disp(a == b)`.
- **Indirect calls through function handles to class methods.** Rare
  in current tests. If it comes up, fall back to the flat function for
  that callee specifically (don't add it to `ClassMethodFuncs`).
- **`enumeration` blocks** (`class_enum.m`). Currently treated as a
  classdef with constants. Likely needs separate handling — emit C++
  `enum class` plus a value lookup. Out of scope for the first cut;
  fall back to the existing wrapper for `enumeration` classes (detect
  via the absence of `obj_new` calls in the method body).

## Test plan

Stdout goldens (`test/Run/*.stdout`) stay locked — the runtime
behaviour is preserved. The `.cpp` shape goldens at `test/EmitC/`
will change for any classdef test:

- `class_basic.cpp.expected`
- `class_dependent.cpp.expected`
- `class_enum.cpp.expected` (carve-out: see Edge cases)
- `class_inherit.cpp.expected`
- `class_operators.cpp.expected`

Plus the operator-overload variants and possibly any cell / struct
tests that incidentally exercise `obj_*` runtime calls. Regenerate via
`UPDATE=1 test/EmitC/run_tests.sh build/matlabc` after hand-eyeballing
that the shape changes match the design.

After the diff lands:

```bash
ctest --test-dir build --output-on-failure
```

All 9 lanes must pass. The `.skip-emit-python` markers stay as-is —
the Python lane already has the equivalent translation from `5adf4e4`.

## Estimated scope

- Lines of new C++ code: ~350 (across Phases 1, 2, 3, 4, 6, 7 in
  EmitC.cpp; Phase 5 is 5 lines)
- Test goldens to regenerate: 5–7 `.cpp.expected` files at `test/EmitC/`
- Wallclock: 1 focused session (4–6 hours)
- Risk areas: Phase 6 type tracking through alloca / store / load
  chains, especially when a slot survives Mem2RegLite (cross-block
  uses); Phase 7 variable-declaration emission must coordinate with
  the existing slot-promotion / DirectSlots logic.

## Reference

Phase-equivalent work for the Python backend already shipped:

- `e01be51` — phase A: rewrite `obj_get/set_f64` to attribute access.
- `5adf4e4` — phase B + C: emit Python `class` blocks, rewrite call
  sites in main / non-class code.
- `runtime/matlab_runtime.py::_MatObj` — the runtime-side glue that
  lets the rewrites round-trip through any non-rewritten call path.

The C++ version is structurally similar but adds: real member-field
declarations (Python uses dynamic attributes), explicit type tracking
for `BankAccount acc(...)` declarations (Python is dynamically typed),
and `: Init(arg)` member-init-list synthesis (Python doesn't need
this).
